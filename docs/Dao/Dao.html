[line 113]

This abstract base class for all Daos.

IMPORTANT NOTE: The Dao (and Record for that matter) depend on the resource  having a primary `id` attribute!  If your resource layout does not provide this attribute, the framework will not work.
The typical usage of the Dao is as follows:  
  1.  <?php
    2. 

  2.    $userDao = new UserDao($datasource); // UserDao extends MysqlDao for example
    3. 

  3.    $userDao->get()->set('username', 'user2000')->save(); // Insert new user
    4. 

  4.    $userDao->getById(4)->set('name', 'New Name')->save(); // Update existing user
    5. 

  5.    $userDao->getByUsername('user2000')->delete(); // Deletes the user.
    6. 

  6.    // getByUsername() has to be implemented by the UserDao
    7. 

  7.  ?>
    8. 

To create a Dao for you resource, simply extend a Dao implementation (eg: MysqlDao)  and set the necessary properties.  Example:  
  1.  <?php
    2. 

  2.    class UserDao extends MysqlDao {
    3. 

  3.  
    4. 

  4.      protected $resourceName = 'users';
    5. 

  5.  
    6. 

  6.      protected $attributes = array(
    7. 

  7.        'id'=>Dao::INT,
    8. 

  8.        'username'=>Dao::STRING,
    9. 

  9.        'status'=>array('online', 'offline'), // This is an enum.
    10. 

  10.        'comment'=>Dao::TEXT,
    11. 

  11.        'creationTime'=>Dao::TIMESTAMP
    12. 

  12.      );
    13. 

  13.  
    14. 

  14.      protected $nullAttributes = array('comment');
    15. 

  15.  
    16. 

  16.      protected $defaultSort = array('creationTime'=>Dao::DESC, 'username'=>Dao::ASC);
    17. 

  17.  
    18. 

  18.      public function getByUsername($username) {
    19. 

  19.        // No checking for SQL Injections has to be done here, since get() will do all of
    20. 

  20.        // that (including checking the type of the attribute username)
    21. 

  21.        return $this->get(array('username'=>$username));
    22. 

  22.      }
    23. 

  23.    }
    24. 

  24.  ?>
    25.

$additionalAttributes = array()

	
[line 237]

  This works exactly the same as the attributes, except that it only defines attributes, that may additionally be returned by the  datasource (for example in joins).

Those values can *not* be set in the Records afterwards, but are checked for their types when retrieved.  When trying to get an additional attribute out of a Record, that has not been retrieved from the datasource, the Record should  just return null and not error.  When trying to set an additional attribute the Record should trigger an error.

$attributeImportMapping = array()

	
[line 327]

  If your datasource holds different attribute names than you want in you Records, you can specify export & import mappings.

(Don't confuse this with your php values. Meaning: if the datasource value is 'strangely_named', but you want to access your record  like this: $record->perfectName, then you have to map it from strangely_named to perfect_name (not perfectName).)  E.g.: array('what_the_datasource_value_actually_is'=>'what_you_would_want_it_to_be');
This variable is called attribute import mapping, but is used to export the attributes too.

$defaultValueAttributes = array('id')

	
[line 347]

  This is a list of attributes that have default values in the datasource.

This means that, if the values are NULL, and the entry is inserted in the datasource, they will not be  passed, so that the datasource can automatically set the values.  So when you call getRawRecord() those values will be null.  Per default the id is given.
